監視範囲が違う2つのアプローチ
tracingは「動いた処理」だけ見える。
exit-code契約は「動かなかった処理」を拾う。

tracing / observability

対象: 実行された処理
span、token、latency、retry。エージェントが通った経路の可視化に強い。
  • タスクごとのtoken消費
  • ツール呼び出しのlatency
  • リトライ / エラー率
  • 起動すらしなかったプロセス
  • 発火しなかったcron
  • 何もせず exit 0 のエージェント

exit-code契約 + heartbeat

対象: 動くべきだった処理
各エージェントが「N時間ごとに成功pingを送る」と宣言。ping切れ = ページャ。
  • そもそも動いたか?
  • 正常終了したか?
  • exit 0 / 1 / 2 を原因にマップ
  • 24h heartbeat切れ → dead man's switch
  • cron wrapperが exit code を記録
  • healthchecks.io / systemd OnFailure=
observabilityは可視性 · 契約は責任 · 両方が要る
