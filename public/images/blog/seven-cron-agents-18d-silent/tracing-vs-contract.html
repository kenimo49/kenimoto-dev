two scopes, two failure modes
Tracing sees what ran. Exit-code contracts see what didn't.

Tracing & observability

scope: things that executed
Spans, tokens, latency, retries. Great for code paths the agent actually ran through.
  • token spend per task
  • tool-call latency
  • retry / error rate
  • a process that never spawned
  • a cron that didn't fire
  • an agent that exited 0 with no work done

Exit-code contract + heartbeat

scope: things that should have run
Each agent declares "I will ping success every N hours." Missing ping = page.
  • did this run at all?
  • did it finish successfully?
  • exit 0 / 1 / 2 mapped to causes
  • 24h heartbeat → dead man's switch
  • cron wrapper logs exit code
  • healthchecks.io or systemd OnFailure=
observability adds visibility · contracts add accountability · you need both
